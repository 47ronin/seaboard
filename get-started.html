Open Data is a valuable resource meant for everyone. But let’s face it – data is not easy to use, and it's even harder to use correctly. Our goal is to empower all San Diego Residents, from the nerdiest to the coolest, to get and use the City’s data. We do this by trying to anticipate and remove common roadblocks to using data that we have run into as data users ourselves.

Exploring Datasets

To see which datasets the City is currently publishing, click on the “Datasets” menu tab. You will see a search interface:

Here, you can begin browsing datasets by using the portal’s search and filter features. If you don’t enter a search term and don’t click on a filter, you will be able to browse the full list of datasets.

Available search filters:
  • Topics - High-level grouping categories.
  • Tags- Secondary grouping categories. A dataset can have multiple tags.
  • Format - the format the files are available in. More information about formats [somewhere else]
  • Publisher - the publishing department

Let's now discuss how we organize the data on our portal.

How the Portal is Organized

To organize our data, we follow DKAN’s Catalog > Dataset > Resource model:

Catalog

The portal’s dataset listing is the catalog.

Dataset

A dataset contains individual resources as well as metadata. Metadata is the “Who, What, When, Where, Why” of each dataset. There are a number of specifications for dataset metadata, but we currently comply with the DCAT and Project Open Data schemas.

We use a dataset as a bucket for multiple views and formats of a single set of data, as well as a place to provide data dictionaries. Each of those permutations and formats is a resource.

Resources

Resources are the actual files, APIs or links that we are sharing. We only provide resources that are machine-readable and in an open format. Resource types include:

  • csv
  • json
  • geojson
  • zip
  • txt
  • xml
  • API

We use resources to provide multiple permutations of data for easiest use on the portal. These permutations may include differences in format or granularity. Let’s walk through an example:

We consider parking meter transactions to be one dataset.

The raw data includes every single transaction for every single parking meter in the city from the beginning of 2015 until today. In total, that’s more than 1 GB worth of data. Excel has a 1,000,000 row limit, so it’s impossible to open in Excel. That means the average user would not be able to use that data effectively.

We split the large file into into 2015 transactions and 2016 transactions. This will make it easier for developers to use, but since the file size for 2015 is still around 500MB, an average Excel user won’t be able to work with these files either.

The non-technical user is probably not interested in transaction level data. He or she likely wants to see just the daily or monthly trend of parking transaction counts or total money made by meters in their neighborhood. We decided to provide pre-made aggregations of the data. For both 2015 and 2016, we roll up the data into totals by month or totals by day.

These totals are the same data - we are just offering the convenience of using the different views rather than the large raw files. We consider the raw files plus the aggregated views to still be one dataset, whereas other portals may consider those to be different datasets.
